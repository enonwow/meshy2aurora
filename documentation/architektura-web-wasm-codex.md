## 2. Product flow

```text
GitHub Pages static files
  -> browser loads UI and WASM
  -> user-selected GLB/MDL/HAK file
  -> React UI
  -> Web Worker
  -> m2a-wasm (Rust/WASM boundary)
  -> m2a-core (pure parsing/conversion/validation)
  -> JSON diagnostics + canonical IR + generated bytes
  -> Three.js preview and report panels
  -> downloadable .hak, JSON reports, and proof manifest
  -> manual NWN EE Toolset/game proof
```

The Worker owns conversion and packaging. The UI thread owns interaction and Three.js rendering. The Worker must never produce a second, divergent conversion implementation.

GitHub Pages serves only the static application. It receives no model bytes, performs no conversion, stores no project state and has no API endpoint in MVP.

## 4. Browser file contract

```yaml
input:
  source_model: [".glb"]
  reference_input_modes: ["HAK package", "loose selected files"]
  accepted_mvp: [".glb", ".mdl", ".mdx", ".hak", ".2da", ".tga", ".txi", ".utc"]
  deferred: [".fbx"]
  base_table_rule: "when the target requires an existing 2DA, the user selects either one HAK containing it or the loose .2da file; the app reads, updates and packages the derived output automatically"
  default: "<input type=file> and drag/drop"
  enhanced: "File System Access API after explicit user gesture and permission"
  forbidden:
    - "reading a local path without user selection"
    - "silent upload of a selected model"
    - "automatic access to the NWN installation"
output:
  default: "Blob + browser download"
  enhanced: "save picker where supported and user-initiated"
  reports:
    - "inspection.json"
    - "validation.json"
    - "conversion-manifest.json"
    - "readback-diff.json when M4 exists"
    - "diagnostics.md"
  generated_game_content: ["MDL/MDX policy", "TGA/TXI", "updated 2DA", "UTC for creature", "GIT proof instance", "HAK"]
```

Status: POTWIERDZONE. The standard File API gives a web application access to files selected by the user; richer filesystem access requires user permission. Sources: https://developer.mozilla.org/en-US/docs/Web/API/File_API and https://developer.mozilla.org/en-US/docs/Web/API/File_System_API

`HAK` is optional as an input container. It is not optional merely because it is convenient: a user may select loose `appearance.2da`, `MDL`, `MDX`, texture and template files directly. The output of the package workflow remains a newly generated HAK; the application never modifies a selected input HAK or loose file in place.

## 5. WASM boundary

```yaml
status: POTWIERDZONE
binding: "wasm-bindgen"
core_api_examples:
  - "inspect_binary_mdl(bytes) -> InspectionReport"
  - "read_hak(bytes, selector) -> HakInspectionReport"
  - "convert_glb(bytes, settings) -> ConversionResult"
  - "pack_hak(resources) -> generated_hak_bytes"
adapter_rules:
  - "bytes cross the JS/WASM boundary once per operation"
  - "reports have schema_version and deterministic ordering"
  - "large operations run in a Web Worker"
  - "UI never interprets MDL offsets or writes HAK bytes"
```

`wasm-bindgen` is the approved bridge because it is designed for high-level Rust/WebAssembly and JavaScript interaction. Source: https://wasm-bindgen.github.io/wasm-bindgen/

## 5A. Reference-resource selection in the browser

```yaml
status: "POTWIERDZONE for browser file selection and ERF/HAK V1.0 structure; Aurora First required for final table-resolution behavior"
canonical_representation: "ResourceIndex keyed by (resref, resource type)"
input_modes:
  hak:
    input: "one user-selected .hak File"
    steps:
      - "read the HAK/ERF header into memory"
      - "read key table and resource table"
      - "build ResourceIndex from package entries"
      - "find appearance.2da by resref appearance plus the 2DA resource type"
  loose_files:
    input: "one or more user-selected files or a user-selected directory"
    steps:
      - "derive resource identity from each selected filename and confirmed type mapping"
      - "build the same ResourceIndex from loose entries"
      - "use directly selected appearance.2da when it is needed"
common_steps:
  - "decode only the required payload into the relevant parser"
  - "show inventory, source mode and selected resources in the UI"
  - "fail explicitly on duplicate or ambiguous (resref, resource type) entries; never guess"
  - "write a derived output; never modify selected input files"
browser_rule: "no extraction to a server or to a local filesystem path is needed"
```

Both routes end at the same `ResourceIndex`, so all later reader, validator and writer code is shared. In HAK mode the application selects resources inside the package. In loose-file mode it uses exactly the files selected by the user. This is an input choice only; it does not change the generated HAK/package route.
